# Truffle — Real Chicago Data Ingestion (Tier B)

**Date:** 2026-06-14
**Status:** Approved design, pending implementation plan
**Author:** David Hsi (with Claude)

## Problem & goal

Truffle today runs on a hand-written mock array (`src/lib/data.ts`) — no backend,
no database, all client-side. The goal is to make it **functional and valuable on
real data**: a genuine, refreshable dataset of Chicago restaurants that powers the
existing "hidden gem" recommender, so a real user in Chicago would find it useful.

This is a **Tier B** effort: a *repeatable build-time ingestion pipeline* that
produces a committed JSON dataset. No live database, no accounts, no real-time
freshness. Clean upgrade path to a full backend (Tier C) later if warranted.

## Core insight: real data fits the existing thesis

The product's whole wedge is `gemScore = (rating/10) * (1 - buzz)` — **high quality
+ low mainstream awareness = hidden gem**. Real data maps onto this cleanly:

- **`rating`** comes from a real star rating.
- **`buzz`** (mainstream awareness) is proxied by **review count**. A spot with 4.7★
  and 80 reviews is undiscovered; 4.7★ and 9,000 reviews is a tourist mainstay.

Because the data *shape* (`Restaurant`) does not change — only its *contents* — the
product's brain (`recommend.ts`, `ranking.ts`, `gemScore`, `store.ts`) stays
**completely unchanged**. We swap what fills `RESTAURANTS`, not its interface.

## Decisions (locked)

| Decision | Choice |
|---|---|
| Ambition tier | **B** — repeatable build-time ingestion → committed JSON, no live DB |
| City | **Chicago** |
| Data source | **Google Places API** (key with billing) |
| Gem signal | `rating` = stars×2; `buzz` = min-max normalized `log(reviewCount)` across the city |
| Editorial fields | **Generated by Claude (Haiku)** at ingest, grounded in real review snippets |
| Photos | **Real Google photos via a server-side proxy route** (`/api/photo`), key never client-side |
| Distance | **Dropped as static**; store lat/lng, compute client-side via geolocation, else show neighborhood |
| Data delivery | **Build-time JSON import** (`src/lib/restaurants.generated.json`) — zero consumer churn |

## Architecture & data flow

```
scripts/ingest.ts  (run on-demand via `npx tsx`, NOT during deploy)
   │
   ├─ Google Places API ──► candidates per neighborhood (Text Search)
   │                         + Place Details (field-masked): rating, userRatingCount,
   │                           priceLevel, types, location, hours, photos, top reviews
   │
   ├─ derive gem fields (rating, buzz, popularity, price, cuisines, lat/lng)
   │
   ├─ Claude (Haiku) ──► { blurb, insiderTip, signatureDishes, vibes[], tags[] }
   │                      grounded in real review snippets
   │
   └─► writes  src/lib/restaurants.generated.json   ──► committed to git
                          │
        data.ts imports it at build time  (RESTAURANTS stays a synchronous array)
                          │
   recommend.ts · ranking.ts · gemScore · store.ts  ── ALL UNCHANGED
                          │
        photos render via  /api/photo?ref=…  (server-side key, cached)
```

User state stays in `localStorage` exactly as today.

## Components

### 1. Ingestion script — `scripts/ingest.ts`

Standalone TypeScript run via `npx tsx scripts/ingest.ts`, deliberately **outside**
the Next build so it never runs on deploy (cost control). Pipeline per neighborhood:

1. **Search** — Places **Text Search** (`places:searchText`) e.g. `"restaurants in
   Logan Square, Chicago"`. Page via `nextPageToken`. Cap **40–60** per neighborhood.
2. **Details** — Place Details with a field mask covering: `displayName`, `rating`,
   `userRatingCount`, `priceLevel`, `types`, `location`, `formattedAddress`,
   `regularOpeningHours`, `photos`, `reviews`, `editorialSummary`, `websiteUri`.
3. **Filter** — keep only food places (types ∩ {restaurant, cafe, bakery, bar, …}),
   drop permanently closed, drop `userRatingCount < 10` (noise), require a `rating`.
4. **Derive** (deterministic, see §2).
5. **Editorial** — one Claude Haiku call per spot (see §3).
6. **Write** `src/lib/restaurants.generated.json` (array of `Restaurant`).

**Caching:** keep a `scripts/.ingest-cache/` keyed by `placeId` so re-runs skip
unchanged spots (avoids re-paying Places Details + Claude). Cache is git-ignored.

### 2. Field derivation (Places → `Restaurant`)

| `Restaurant` field | Source |
|---|---|
| `id` | slug of name + placeId suffix (stable) |
| `name` | `displayName.text` |
| `rating` (0–10) | `rating` × 2 |
| `buzz` (0–1) | min-max normalized `log(userRatingCount)` over the whole city dataset (computed in a second pass after all spots collected) |
| `popularity` (0–1) | percentile rank of `userRatingCount` across the dataset |
| `price` (1–4) | `priceLevel` enum → 1–4 (default 2 if absent) |
| `cuisines` | map fine-grained Places types (`italian_restaurant`, `ramen_restaurant`, …) → `Cuisine` union; fall back to name heuristics; Claude fills gaps |
| `neighborhood` | the search bucket queried |
| `city` | `"Chicago"` |
| `lat`, `lng` | `location.latitude/longitude` (**new fields**, see §6) |
| `vibes`, `tags`, `signatureDishes`, `dietary`, `spice` | Claude (heuristic seed: price 4 → fine-dining; cafe → casual/quick-bite; bar → late-night) |
| `blurb`, `insiderTip` | Claude |
| `reels` | single entry `[{ poster: "/api/photo?ref=<photoName>", … minimal legacy fields }]` |
| `distanceKm` | **removed from static data** (computed client-side; see §6) |

Two-pass: collect raw spots first, then compute `buzz`/`popularity` once the full
city distribution is known (normalization is relative to the city).

### 3. Editorial generation (Claude Haiku)

One call per restaurant. Input: `{ name, cuisines, price, rating, reviewCount, topReviewSnippets[3–5] }`.
Output (structured JSON, validated): `{ blurb, insiderTip, signatureDishes[], vibes[], tags[], dietary[], spice }`
— `vibes`/`dietary` constrained to the `Vibe`/`Dietary` unions, `spice` to 0–3.
Grounded strictly in the supplied facts/reviews — no invented claims. Model:
`claude-haiku-4-5`. Reuses the existing `ANTHROPIC_API_KEY` convention. Results
cached by `placeId`.

### 4. Photo proxy — `src/app/api/photo/route.ts`

`GET /api/photo?ref=<photoName>` → fetches the photo bytes from the Google Places
**Photo** endpoint using the **server-side** `GOOGLE_PLACES_API_KEY`, streams the
image back with a long `Cache-Control` (e.g. `public, max-age=604800, immutable`)
since hero photos rarely change — this bounds the per-view Places Photo cost.
Validates the `ref` format before calling out. Same-origin, so the API key never
reaches the client and `next.config.mjs` `remotePatterns` needs no change.

### 5. Data delivery — build-time JSON import

`scripts/ingest.ts` writes `src/lib/restaurants.generated.json`. `src/lib/data.ts`
becomes:

```ts
import generated from "./restaurants.generated.json";
import { Restaurant } from "./types";
export const RESTAURANTS = generated as Restaurant[];
```

Next bundles the JSON at build time, so `RESTAURANTS` stays a synchronous array and
**every existing import keeps working unchanged**. (The mock array is removed; the
file may keep the `img()` helper only if still referenced.)

### 6. Dynamic distance (honest change)

Static `distanceKm` was meaningless mock data. We **store `lat`/`lng`** on each
restaurant and compute distance **client-side** only when the user grants
geolocation (haversine vs. their position). When geolocation is unavailable, the UI
shows the **neighborhood** (already present). Touch points: `SpotCard`, the detail
page's "X km away" line. Add optional `lat`/`lng` to the `Restaurant` type; make
`distanceKm` optional/derived.

### 7. Configuration

- `GOOGLE_PLACES_API_KEY` — **server only** (ingest script + photo route). Never `NEXT_PUBLIC_`.
- `ANTHROPIC_API_KEY` — existing convention (ingest editorial + live concierge).
- Neighborhood list lives in the ingest script (or `scripts/neighborhoods.ts`).

### 8. Chicago neighborhoods (first pull, 9)

Logan Square, Pilsen, West Loop, Avondale, Argyle/Uptown, Andersonville,
Bridgeport, Chinatown, **Lakeview** — chosen for food identity *and* gem density.

## Cost

- **Google Places:** Text Search + field-masked Details for ~350–550 spots — a
  one-time pull comfortably inside Google's monthly free credit.
- **Claude Haiku:** ~one small call per spot ≈ single-digit dollars for the city.
- **Ongoing:** only the photo proxy (Places Photo SKU), kept low by aggressive
  caching. No other recurring cost (no DB, no server compute beyond the proxy).

## Explicitly out of scope (YAGNI)

- No live database, no user accounts / auth (localStorage stays).
- No real-time freshness — refresh = re-run the script + commit the new JSON.
- No multi-city.
- No server-side persistence of user data.
- No change to `recommend.ts`, `ranking.ts`, `gemScore`, or `store.ts` logic.

## Verification

Per CLAUDE.md (no test runner by design):
- `npm run typecheck && npm run build` must pass with the generated data.
- Spot-check the generated JSON: gem scores rank sensibly (low-review high-rating
  spots float up), no nulls in required fields, neighborhoods balanced.
- Manual 375px visual pass on feed / detail / search with real data + real photos.
- Verify the photo proxy returns images and the API key is absent from the client bundle.
