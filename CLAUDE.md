# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Truffle** — a calm, editorial discovery app for under-the-radar restaurants (Next.js 14 App Router + TypeScript + Tailwind + Zustand). It surfaces high-quality, low-buzz "hidden gems" ranked for the user's taste. It was rebranded and redesigned from a prior dark, TikTok-style "ReelEats" reels feed; that surface is fully retired (see `docs/superpowers/specs/2026-06-13-truffle-redesign-design.md`).

## Commands

```bash
npm run dev        # local dev at http://localhost:3000
npm run build      # production build (this is a primary correctness gate)
npm run start      # serve the production build
npm run typecheck  # tsc --noEmit (strict)
npm run lint       # next lint
npm run ingest         # regenerate src/lib/restaurants.generated.json (full) from Google Places
                       # AND derive src/lib/restaurants.core.json (client-safe subset).
                       # (needs GOOGLE_PLACES_API_KEY in .env; ANTHROPIC_API_KEY optional).
                       # Add `-- --sample` for an offline 2-record run from the fixture.
npm run split-data     # re-derive restaurants.core.json from the full generated dataset
                       # (also run automatically at the end of `npm run ingest`)
npm run enrich-dishes  # Ordering Phase 3: add ONLY dishDescriptions to the existing
                       # generated dataset (Claude, needs ANTHROPIC_API_KEY), then
                       # re-derive core. Idempotent; `-- --limit N` for a cheap subset.
                       # Surgical — does NOT re-pull Places or churn other editorial.
npm run validate-data  # assert invariants on the generated dataset
```

**There is no test runner** (no jest/vitest, by design — this is a presentation-heavy app over deterministic logic). The verification gate for any change is: `npm run typecheck && npm run build`, plus targeted `grep` for regressions and a manual visual pass at a 375px viewport. Do not add a test framework unless asked.

## Architecture (the big picture)

The app is almost entirely **client-side**. There is no database and no auth. Restaurant data is a **generated, committed JSON dataset** (real Chicago spots from the Google Places API — see Data pipeline below); all user state lives in the browser.

- **Data is split into a client `core` and a server-only `full` dataset.** `src/lib/restaurants.generated.json` is the **full** source of truth (every field). `scripts/split-data.ts` derives `src/lib/restaurants.core.json` = the full dataset **minus the detail-only fields `insiderTip`, `blurb`, `hours`, and `dishDescriptions`** (`DETAIL_ONLY_FIELDS`) — those are ~25%+ of the payload and only the detail page / server concierge use them. Both files are committed; **do not hand-edit either — regenerate with `npm run ingest`** (or `npm run split-data` to re-derive only core).
- **`src/lib/data.ts`** (client-safe) — exports `RESTAURANTS` by importing `restaurants.core.json`, plus `RESTAURANTS_BY_ID` (O(1) id index), `getRestaurant`, and `ALL_CUISINES`. Records here have **no `insiderTip`/`blurb`/`hours`** (all optional on the `Restaurant` type).
- **`src/lib/data.server.ts`** (server-only) — imports the **full** `restaurants.generated.json` and exposes `RESTAURANTS_FULL` + `getFullRestaurant(id)`. **Never import this from a client component** (it re-bundles the full dataset). This is the single seam a future DB would replace — swap the JSON read for a query and callers stay the same. Used by the detail page (server component) and the assistant route.
- Each `Restaurant` carries `rating` (Google stars ×2), `popularity`, `buzz` (mainstream awareness), `lat`/`lng`, `insiderTip` (detail-only), and `reels[]` (only `reels[0].poster` — a `/api/photo?ref=…` proxy URL — is used by the UI; the video/emoji/gradient reel fields are legacy, not rendered).
- **`src/lib/types.ts`** — domain types plus `gemScore(r) = (rating/10) * (1 - buzz)`. **Low `buzz` = hidden gem.** This score drives the feed, search, and badges. Anything claiming "earliness"/"under the radar" derives from `buzz` — there is no historical save-count data, so never imply live counts.
- **`src/lib/recommend.ts`** — the core. `recommend(state)` and `scoreRestaurant(r, state)` are a content-based, **explainable** scorer: every signal pushes a human-readable `reason`. State is `{ profile, liked, saved, ranked, seen, neighborhood, neighborhoodStrict, openNow, nowMs }`. `parseQuery()` is the NL search/assistant parser — it extracts a **neighborhood** name (matched against the dataset's real neighborhoods) and an **"open now" intent**. **Open-now steer** (concierge-only): when `openNow`+`nowMs` are set, the scorer boosts spots `isOpenNow` (`src/lib/hours.ts`, judged against each venue's own `utcOffsetMinutes`) and demotes closed ones — a steer, never a hard filter, and `unknown` hours are never penalized. Inert on the client (client `core` carries no `hours`). Both the Discover home's search mode and the AI concierge feed that into the scorer: a named area gets a **strong steer** (`neighborhoodStrict`, +40 in-area + a mild out-of-area demotion) vs. the feed's gentle ambient nudge; it's never a hard filter, so a thin area falls back to the closest nearby spots rather than emptying. This is the product's brain — keep it intact; redesign work is presentation only.
- **`src/lib/ranking.ts`** — Beli-style pairwise-comparison ranking (binary-search insertion) used by `RankModal` to place a "been" spot on the 0–10 leaderboard.
- **Dish top-3 (one section, within one restaurant — never cross-restaurant, never live votes).** The detail page has a **single** dish section, `OrderGuide` ("What to order"): taste-aware picks from the restaurant's real `signatureDishes`, each with a "why it fits you", an optional **crowd note** ("★ Reviewers love it for …"), and an allergen caution. The crowd note comes from `Restaurant.topDishes` — a ranked ≤3 subset of `signatureDishes`, each with a review-grounded `note`, distilled at ingest by Claude Haiku (`scripts/editorial.ts`, validated ⊆ `signatureDishes` in `coerce()` + `validate-data.ts`), **kept in the client `core`** (small + display-facing). When present, `topDishes` also sets the picks' order (`order.ts` prefers it) and `note` rides on each `OrderPick` (so the crowd signal shows inline, not as a duplicate section). `topDishes` is optional — empty until a keyed `npm run ingest` populates it, and the note simply doesn't render until then (honest provenance, **no counts**). **A personal per-dish ranking and live cross-user voting were both considered and dropped** — over a ≤4-dish pool a personal pairwise ranker is ceremony-without-payload and clashes with the calm ethos, and voting trips the deferred-DB trigger (per-dish cold-start would be dishonest on hidden gems).
- **`src/lib/store.ts`** — Zustand store persisted to `localStorage` under key `truffle-store`. Holds `profile, liked, saved, ranked, seen, onboarded`. User interactions feed back into `recommend()`, so the feed personalizes over time.
- **`src/components/AppShell.tsx`** — wraps every screen in a phone frame, gates on `onboarded` (redirects to `/onboarding`), and renders `BottomNav`. Pages opt out of nav via props.
- **Discover home** (`/feed`; merged Feed + Search 2026-06-19 — design: `docs/superpowers/specs/2026-06-19-discover-home-design.md`, decision: `docs/decisions/2026-06-19-discover-home-merge.md`) — one editorial surface with two modes off the store's `searchSubmitted`/`searchCuisine`. **Browse mode** = a sticky `SearchBar` + `NeighborhoodChips` + a **hero** ("Tonight's gem") + `HelpMeDecide` + personalized **carousels** + a paginated **tail** ("More to discover"). Carousels come from **`src/lib/shelves.ts` `buildShelves()`** — a *pure composer* that slices the single `recommend()` pass / `gemScore` into themed rows (`In {neighborhood}`, `Because you saved {X}`, `More {cuisine} you'd love`, `Under the radar`), each spot claimed by at most one shelf, remainder → tail; it adds **no new scoring**. Shelf UI is `src/components/discover/{SearchBar,Shelf,ShelfCard,HeroSpot}.tsx`; `ShelfCard` is the compact, `React.memo`'d carousel cousin of `SpotCard`. **Search mode** (query submitted *or* cuisine selected) swaps the shelves for the prior search-results behavior (literal `matches()` + `parseQuery()` + `recommend()` on a filtered pool, honest `fallback`/`nearby` headers, and the elevated concierge hand-off for conversational queries). The standalone `/search` route now just `redirect("/feed")`; bottom nav is 4 tabs (Discover · Map · AI · You).
- **Feed model:** the feed is an **editorial list of one `SpotCard` per restaurant** (not per reel). `SpotCard` is deliberately minimal (photo · gem score · name · one metadata line · save) and **`React.memo`'d** (it re-renders on every store mutation; without memo a save toggle re-renders every card). `src/components/Feed.tsx` **paginates** (renders 24, "Show more" reveals more) so we never mount ~1.6k cards / fire ~1.6k photo requests at once; the "you're caught up" end state shows once everything is revealed — it now renders the Discover home's deep-browse **tail**. Card images use `loading="lazy"`. **All depth lives on the detail page**: the "why you" reasons (recomputed via `scoreRestaurant`), insider tip, earliness cue, and full actions including `ShareSpot`. Preserve this progressive-disclosure split — do not re-clutter the card.
- **Detail page** (`src/app/restaurant/[id]/page.tsx`) — a **server component** that loads the full record via `getFullRestaurant(id)` (so detail-only editorial arrives per-record, not in the client bundle) and hands it to `src/components/RestaurantDetail.tsx` (the client component with the interactive UI + client-side `scoreRestaurant`). It also exports `generateMetadata` (per-restaurant title/description) and an `opengraph-image.tsx` route; unknown ids `notFound()`.
- **AI concierge** (`src/app/api/assistant/route.ts`) — builds a candidate pool with the local engine over the **full** dataset (`RESTAURANTS_FULL`, so candidates keep `insiderTip`/`blurb`), then calls Claude **if `ANTHROPIC_API_KEY` is set** (`ANTHROPIC_MODEL` optional), else returns a deterministic local result. The feature must always work without a key. A best-effort per-IP rate limit (15/min, in-memory → per-instance under Fluid Compute) caps Claude spend; returns 429 with a friendly `reply`. It also fields **ordering questions**: if the query reads like "what should I order at X" *and* names a real restaurant, it short-circuits to the ordering engine (below) and returns a dish guide instead of recommending spots.
- **Ordering guide / "what to order"** — `src/lib/order.ts` is a **pure, client-safe** module (`buildLocalOrderGuide`, `sanitizePicks`, `orderGuideToReply`): it turns a restaurant's real `signatureDishes` + the user's `TasteProfile` into 2–3 taste-aware picks (it **never invents a dish** — `sanitizePicks` validates picks against `signatureDishes`). `src/lib/order.server.ts` (`askClaudeOrder`) is the server-only Claude upgrade, mirroring the assistant's key-less-fallback pattern; it now powers the **concierge** "what to order at X" path (`/api/assistant`). On the detail page, **`src/components/OrderGuide.tsx`** renders **entirely from the full record the page already holds — no network, no skeleton, no reorder** (a collapsed accordion: tap a dish to expand its description). The rich, dish-centric copy is **pre-stored at ingest** in `Restaurant.dishDescriptions` (detail-only; generated by `generateDishDescriptions` in `scripts/editorial.ts`, populated by `npm run enrich-dishes`) and renders as the dish's description; the local taste line (`tasteWhy`) is now a **fallback only**, shown when a dish has no stored description — so the guide is instant and stable. **The old request-time `/api/order` upgrade was removed from the detail page** (it added a ~6–7s delay and visibly reshuffled the picks after load); `src/app/api/order/route.ts` still exists but is no longer called by the UI. Rationale + the deferred "Option B" (keep live, but lock the order): `docs/decisions/2026-06-21-prestored-dish-descriptions.md`. **Allergen cautions (Phase 2):** `TasteProfile.allergies` (the US "big 9", captured in onboarding, distinct from `dietary`) drives per-pick "may contain — ask the kitchen" cautions, computed **request-time** — a conservative dish-name keyword scan (`dishAllergenFlags` in `order.ts`, works keyless) **unioned** with Claude's read when a key is set. These only ever *add* a caution and never claim a dish is allergen-free (keyword matching can't see hidden ingredients), so the "confirm with the restaurant" note is always shown. The recommendation engine is intentionally **not** allergy-filtered.
- **Photo proxy** (`src/app/api/photo/route.ts`) — server-only route (`runtime = "nodejs"`) that streams a Google Places photo using `GOOGLE_PLACES_API_KEY`; the key is **never** client-exposed. Generated posters point at `/api/photo?ref=<encoded photoName>`; responses are hard-cached.
- **Dynamic distance** — restaurants store `lat`/`lng`; `src/components/UserDistance.tsx` (client) shows live distance from browser geolocation on the detail page, falling back to just the neighborhood. The static `distanceKm` (distance from the Loop) is kept only so `recommend.ts`'s proximity nudge stays unchanged — never surface it as a real user distance.
- **Personal Map** (added 2026-06-19; design: `docs/superpowers/specs/2026-06-19-personal-map-design.md`) — a nav tab `/map` (`src/app/map/page.tsx`) that **dynamically imports `MapView` with `ssr:false`** (Leaflet touches `window` at load; the split also keeps Leaflet + the dataset out of the route's First Load JS). `src/components/MapView.tsx` is a **Leaflet + react-leaflet** surface on the free **CARTO "Positron" light** basemap (no API key — the existing `GOOGLE_PLACES_API_KEY` is Places-only). It plots a **small personal pin set** (`buildPins`): your **been** (ranked, shows your 0–10), **saved**, then the top ~40 **nearby gems** (nearest 150 by `haversineKm` from the center, then by `gemScore`) — deliberately not all ~1.6k spots, so **no clustering**. Markers are `L.divIcon` with **Tailwind token classes** (Leaflet renders marker HTML outside React but global utilities still apply), so the CARTO tiles are the **only** third-party-styled surface. Geolocation centers on the user (gentle one-shot like `UserDistance`), falling back to the neighborhood centroid then `CHICAGO_CENTER` (in `geo.ts`); tapping a pin raises a token-styled **bottom card** (not a Leaflet popup) with a save toggle and a link to detail. ("Open now" concierge awareness was scoped out — it needs a hours re-ingest; seams are in the spec.)
- **Launch surfaces** (added 2026-06-19; design: `docs/superpowers/specs/2026-06-19-launch-hardening-design.md`) — **Analytics:** `src/lib/analytics.ts` is a typed `track()` over Vercel Analytics; use it for new funnel events (don't call `@vercel/analytics` directly), and keep the `AnalyticsEvent` union authoritative. `<Analytics/>`+`<SpeedInsights/>` live in `layout.tsx`. **Errors:** `error.tsx`/`global-error.tsx`/`not-found.tsx` are branded; the detail route uses `notFound()` for bad IDs. **SEO/OG:** the detail route is split into a server `page.tsx` (`generateMetadata`) + client `RestaurantDetail.tsx`; `opengraph-image.tsx` renders a per-restaurant card with the real photo (inlined as a data URI). `src/lib/site.ts` holds `SITE_URL` — set `NEXT_PUBLIC_SITE_URL` in prod for canonical/OG URLs.

## Data pipeline (regenerating restaurant data)

`scripts/ingest.ts` (run via `npm run ingest`, **never** during build) pulls restaurants from the Google Places API (New) across the neighborhoods in `scripts/neighborhoods.ts` — each runs a generic "restaurants in X" query plus cuisine-targeted probes (`neighborhoodQueries()` × `CUISINE_PROBES`) to surface the long tail a generic search caps out before reaching; overlapping results are deduped by placeId. It then derives domain fields (`scripts/derive.ts` — incl. `buzz` = min-max normalized `log(reviewCount)` across the city, and `cuisinesFromTypes`), generates editorial copy **and** cuisines with Claude Haiku (`scripts/editorial.ts`; preferred over the type-derived cuisine, and it falls back to a deterministic template without `ANTHROPIC_API_KEY`), caches by placeId (`scripts/cache.ts`), writes `src/lib/restaurants.generated.json`, and finally calls `scripts/split-data.ts` to derive the client-safe `src/lib/restaurants.core.json` (full minus `insiderTip`/`blurb`). `scripts/validate-data.ts` then asserts invariants on the generated dataset.

- **Keys** live in `.env` (gitignored, never committed): `GOOGLE_PLACES_API_KEY` (required) + `ANTHROPIC_API_KEY` (optional). On Vercel they're set as Production env vars (plus `NEXT_PUBLIC_SITE_URL`). To regenerate editorial, `rm -rf scripts/.ingest-cache` first — it's keyed by placeId and silently reuses stale copy otherwise.
- **Deployment:** the repo is connected to a Vercel project (Git integration) — **every push to `main` auto-deploys to Production**; PRs/branches get Preview deploys. Live at `https://food-app-ecru-ten.vercel.app`. Full operational truth (URL, env vars, verify steps, custom-domain how-to) is in **`docs/deployment.md`** — read it before touching deploy/env config.
- **`scripts/` must stay CommonJS** — do NOT add `scripts/package.json` with `"type":"module"`. Under tsx + Node 25 an ESM script can't named-import from a `src/` `.ts` (CJS-default) file ("does not provide an export named X"), which breaks the whole `scripts/ → ../src/lib/*` import chain.
- **Verifying scripts:** use committed-then-deleted static-import `.ts` probes run via `npx tsx scripts/<x>.ts`. Do NOT verify with `npx tsx -e "import('./...')"` — dynamic import wraps named exports under CJS and gives false failures.
- `scripts/*.check.ts` are `node:assert`-based checks for the pure logic (`npx tsx scripts/derive.check.ts`, `editorial.check.ts`); they are not wired into an npm script, consistent with the no-test-runner convention.

## Design system & conventions (enforced)

The look is **"Warm Editorial"** — calm/curated, intentionally NOT a doomscroll, casino, or dating-swipe app. When editing UI, follow these (they are actively swept for):

- **Semantic color tokens only, no raw hex in components.** Tokens in `tailwind.config.ts`: `paper`, `paper-raised`, `ink`, `ink-soft`, `ink-faint`, `olive`, `olive-deep`, `line`, `gem`. (The old dark `text-white/*`, `bg-white/*`, `bg-zinc`, `brand` classes are gone — don't reintroduce them.)
- **Fonts:** `font-display` = Fraunces, `font-sans` = Inter, loaded via `next/font/google` in `layout.tsx`. Headings use `font-display font-semibold` (not `font-black`).
- **No emoji as UI.** Use the SVG icon family in `src/components/icons.tsx`. Text glyphs `◆ ★ ◷ ·` are acceptable accents; pictographic emoji (💎 🔥 ✨ faces) are not. Emoji inside `.replace()` regexes (stripping them out of data labels) are fine.
- **Images:** SpotCard / detail / RankModal / profile use a raw `<img>` with `// eslint-disable-next-line @next/next/no-img-element`. Real photos are served **same-origin** via the `/api/photo` proxy (so they need no `remotePatterns` entry); `next.config.mjs` still whitelists `images.unsplash.com` for any legacy/stock use.
- **No dark patterns:** no autoplay, no infinite supply (show a "you're caught up" end state), no streaks, no slot-machine/spinner mechanics. "Help me decide" must stay a single deterministic pick, never a roulette.
- Respect `prefers-reduced-motion` (handled globally in `globals.css`); keep motion quiet (`animate-floatUp`).
- Don't use "reel"/"reels" in user-facing copy (retired concept); `.reels[]` property access in code is fine.
